# 优缺点

```
┌──────────────────────────────────────────────────────────────┐
│                     跨端动态化容器优缺点成因图                 │
├──────────────────────────────────────────────────────────────┤
│ 1. React Native（RN）                                         │
│   架构：JS线程 + Bridge + Native UI                           │
│   优点成因：                                                   │
│     - 跨端复用率高：JS代码共享 → Bridge调用Native组件          │
│     - 动态更新强：JS Bundle可下发 → 引擎加载新Bundle           │
│   缺点成因：                                                   │
│     - 性能瓶颈：Bridge批量通信 → 异步延迟                      │
│     - 动画差：JS线程计算动画 → 频繁跨线程通信                   │
│     - 首屏慢：引擎初始化 + Bundle解析                          │
├──────────────────────────────────────────────────────────────┤
│ 2. Flutter                                                     │
│   架构：Dart线程 + Skia引擎 + 自绘UI                           │
│   优点成因：                                                   │
│     - 性能接近Native：Dart编译为Native指令 → 直接驱动Skia绘制   │
│     - 跨端一致：自绘UI → 不依赖平台控件                        │
│     - 动画流畅：UI线程与GPU线程协作 → 减少跨线程通信          │
│   缺点成因：                                                   │
│     - 包体大：引擎 + Skia占用数MB                              │
│     - 动态更新难：AOT编译 → 无法直接替换代码                    │
│     - 嵌入复杂：引擎与Native生命周期集成难                     │
├──────────────────────────────────────────────────────────────┤
│ 3. KMP + CMP                                                   │
│   架构：共享Kotlin业务逻辑 + 原生UI渲染                        │
│   优点成因：                                                   │
│     - 逻辑跨端复用：KMP编译成JVM字节码/LLVM bitcode             │
│     - 性能好：UI用原生控件 → 无额外渲染层                       │
│     - 原生体验：直接调用平台UI API                             │
│   缺点成因：                                                   │
│     - UI复用率低：需分别实现Android/iOS UI                     │
│     - 生态小：组件库少                                         │
├──────────────────────────────────────────────────────────────┤
│ 4. WebView                                                     │
│   架构：Native容器 + 浏览器内核                                │
│   优点成因：                                                   │
│     - 实现简单：嵌入系统WebView → 调用loadUrl                   │
│     - 动态更新：HTML/JS可直接从服务端加载                       │
│   缺点成因：                                                   │
│     - 性能差：渲染由浏览器内核完成 → JS↔Native交互慢            │

├──────────────────────────────────────────────────────────────┤

│   架构：平台容器 + 定制WebView或自绘引擎                        │

│     - 无需安装：运行在宿主App                                  │
│     - 动态更新：代码包由平台下发                               │
│   缺点成因：                                                   │
│     - 依赖平台：运行环境受平台控制                             │
```

## RN
### 核心原理
JS + Native Bridge，UI由Native组件渲染
### 优点
跨端复用率高：  
业务逻辑和 UI 用 JavaScript 编写，通过 React 组件描述 UI，运行在 JS 引擎（JavaScriptCore/Hermes）上  
JS 层逻辑通过 Bridge 调用 Native UI 组件，Android/iOS 共享同一套 JS 代码  

动态更新：  
JS Bundle 可通过服务端下发，不需要重新发版  
JS 引擎加载新的 JS Bundle 并重新渲染 UI  

生态成熟、React语法易上手
### 缺点
Bridge通信有延迟：  
JS 与 Native 之间通过异步 JSON 消息队列通信，Bridge 批量传输消息，增加了渲染耗时，频繁调用导致延迟  

复杂动画性能差：  
动画计算在 JS 线程，频繁与 UI 线程通信，JS 线程阻塞或 Bridge 延迟会导致掉帧  

首屏加载慢：  
需要加载 JS 引擎（JS 引擎初始化）和 JS Bundle（Bundle 解析）  
### 适用场景
广告卡片、活动页、落地页、重展示弱交互页面、需要快速迭代的业务
## Flutter
### 核心原理
Dart + Skia引擎，UI完全由Flutter渲染
### 优点
性能接近Native：  
UI 渲染完全由 Flutter 引擎（Skia）绘制，不依赖平台原生控件  
Dart 代码通过 JIT/AOT 编译为 Native 指令，直接驱动 Skia 绘制到 Surface  
2024 年默认启用 Impeller 渲染引擎，解决了之前 Skia 引擎的「首次运行卡顿」问题，iOS 端的性能已经与原生持平，Android 端也大幅改善

跨端一致性好：  
自绘 UI，渲染结果不依赖平台控件，所有控件都是 Flutter widget，跨平台一致  

动画流畅：  
动画计算和渲染在同一引擎内完成  
Dart UI 线程与 GPU 渲染线程协作，减少跨线程通信
### 缺点
包体大：  
需要打包 Flutter 引擎和 Skia  

引擎嵌入复杂：  
Flutter 引擎与 Native 生命周期、路由集成复杂  

动态更新受限：  
Dart AOT 编译成 Native 代码，无法直接替换，需依赖热重载/热更新方案  
### 适用场景
高性能、复杂UI、多端一致性要求高的业务（如电商首页、复杂广告）
## KMP + CMP
### 核心原理
Kotlin多平台共享业务逻辑，UI用Compose多平台
### 优点
共享业务逻辑：  
KMP 允许共享 Kotlin 代码到 Android/iOS  
Kotlin 编译成 JVM 字节码（Android）和 LLVM bitcode（iOS）  

性能好：  
UI 用原生渲染，直接调用平台原生 API，无额外渲染层  
### 缺点
UI跨端复用率低：  
只用KMP进行逻辑复用，UI 层需分别实现 Android/iOS  
引入CMP可以实现UI复用，但内存占用大，存在UI不一致问题，iOS不友好  

生态不如RN/Flutter
### 适用场景
业务逻辑跨端复用（算法、数据层），UI需原生体验的业务
## WebView
### 核心原理
在Native中嵌入浏览器组件
### 优点
实现简单：  
直接嵌入系统 WebView，调用系统浏览器内核渲染 HTML/CSS/JS  

动态更新方便：  
HTML/JS 可直接从服务端加载，WebView 加载 URL 或 HTML 字符串  

生态成熟
### 缺点
性能差（尤其动画、滚动）：  
滚动、动画性能受限于浏览器内核，浏览器渲染管线复杂，帧率不稳定   

原生交互弱：  
渲染由浏览器内核完成，UI 与 Native 交互成本高，JS ↔ Native 通过 JSBridge，延迟高  
### 适用场景
内容页、资讯流、简单广告落地页
## 小程序
### 核心原理
定制化WebView容器 + 框架（微信、支付宝等）
### 优点
生态强：  
依托微信、支付宝、抖音等平台,平台提供运行环境、API、分发入口    

无需安装App:  
运行在宿主 App 内  

动态更新:  
代码包由平台下发
### 缺点
依赖平台:  
运行环境由平台控制  

性能受限:  
渲染基于 WebView 或平台自绘引擎
### 适用场景
平台内运营活动、轻量业务、引流场景
# 业务选择
1.业务迭代速度  
是否需要频繁上线新样式？  
是否需要服务端下发模板动态渲染？  

2.性能要求  
是否有复杂动画、滚动、3D效果？  
是否需要接近 Native 的性能？  

3.跨端一致性  
是否需要 Android/iOS/Web 一致的 UI？  

4.包体大小  
容器是否会显著增加包体？  

5.生态与团队能力  
团队是否熟悉 React、Dart、Kotlin？  

6.平台依赖  
是否依赖微信、支付宝等平台？  

7.用户触达  
用户是否需要安装 App 才能使用？
